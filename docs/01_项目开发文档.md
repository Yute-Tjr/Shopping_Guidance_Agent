## 1. 项目目标与范围

### 1.1 课题定位

把传统"展示型电商广告"升级为"交互型 AI 导购"，在 iOS 原生 App 中让用户通过文本 / 图片 / 语音与 Agent 多轮对话，由后端基于 RAG 在 100 条脱敏商品数据上检索、推理，输出**事实可信、可点击、可加购**的导购回复。

### 1.2 评审权重（必须时刻对齐）

| 维度 | 权重 | 本项目落点 |
| --- | --- | --- |
| 基础功能完整性 | 35% | P0–P2：端到端最小闭环必须先跑通 |
| 工程质量 | 25% | P3：目录结构 / 依赖管理 / 注释 / 错误处理 / 文档齐全 |
| 效果与可靠性 | 20% | P4：检索准确率、无幻觉、复杂语义处理 |
| 加分项深度 | 20% | P5–P6：选 1–2 项做深，不追求广度 |

### 1.3 强制规范（来自课题说明，违者扣分）

- 客户端必须 **iOS 原生（Swift / SwiftUI）**，不接受 Web / H5 套壳。
- 前后端代码分目录：`/client`、`/server`、`/docs`。
- 依赖必须显式声明版本：`requirements.txt`、`Podfile / Package.swift`。
- 核心逻辑（RAG 链路、Prompt 构造）必须有注释。
- **严禁幻觉**：Agent 不得编造库内不存在的商品、价格、SKU、优惠券。所有推荐字段都必须来自检索结果。
- API Key 严禁直接提交到 Git，统一通过 `.env` 注入并写入 `.gitignore`。

### 3.2 一次完整请求时序（文本对话场景）

1. iOS 用户输入 → `POST /chat/stream` (SSE 长连接, body 含 `session_id` + `message`)
2. Gateway 加载会话上下文（最近 N 轮），交给 **Agent Orchestrator**。
3. Orchestrator 调用 **Intent Router**：判断为「商品推荐 / 对比 / 购物车操作 / 闲聊」之一。
4. 若需检索 → **Query Rewriter** 抽取结构化筛选条件（价格区间、品类、品牌排除等），生成 dense query。
5. **Hybrid Retriever** 在 Milvus Lite 上做向量检索 + 元数据过滤，召回 Top-K。
6. **Reranker** 用规则 + LLM 打分（可选）做精排，输出 Top-N（通常 3–5 件）。
7. **Prompt Builder** 把检索片段 + 工具调用结果 + 系统提示拼成 messages，调用 Doubao LLM `stream=True`。
8. LLM 输出按 token 逐步通过 SSE `data:` 帧推送给 iOS；同时把结构化「商品卡片 JSON」作为独立 SSE 事件（`event: product_card`）推送。
9. iOS 端按事件类型分别渲染：`event: token` 逐字追加文本，`event: product_card` 在文字流末尾插入卡片。
10. 流结束发送 `event: done`，客户端把整轮对话写入本地会话存储。

### Phase 1 · 数据工程与向量索引（2 天）

**目标**：100 条商品全部入库，可用关键词检索召回。

- [ ] 实现 `rag/chunker.py`：按 `marketing_description` / 单条 `faq` / 单条 `review` 分别切片，每片附带 `product_id` `chunk_type` `category` `price` 等元数据
- [ ] 实现 `rag/embedder.py`：封装 Doubao-embedding-text 调用，带 batch & 重试
- [ ] 实现 `db/milvus_client.py`：创建 `products` collection，schema 含 `id / product_id / chunk_type / text / vector / metadata(JSON)`
- [ ] 编写 `scripts/seed_mysql.py`：一次性把 100 条商品主表 + SKU 子表灌入 **MySQL**（商品主表 1×100，SKU 表 1×N≈300）
- [ ] 编写 `scripts/build_index.py`：一次性遍历 `ecommerce_agent_dataset/`，生成 chunk → 向量 → 写入 Milvus（**不再**重复写商品主表，主表已在 seed_mysql 灌好）
- [ ] 写单测：随机 5 条 query 验证召回 Top-5 中包含正确 `product_id`

**验收**：`python scripts/seed_mysql.py` 把 100 件商品入 MySQL 成功；`python scripts/build_index.py` 跑通；`python scripts/eval_recall.py` Top-5 recall ≥ 0.8。

> Chunking 策略细节、字段定义、防幻觉约束见 `docs/02_数据工程与RAG设计.md`。

### Phase 4 · 对话能力增强（3 天）

**目标**：覆盖课题"进阶"场景，把得分维度"效果与可靠性 20%"拉满。

- [ ] **多轮上下文记忆**：`agent/memory.py` 保留最近 6 轮，超过自动摘要压缩；session_id 复用
- [ ] **结构化筛选**：在 query rewriter 中抽取 `price_max / price_min / category / brand_exclude` 等字段，作为 Milvus metadata filter 传入
- [ ] **反选与排除**：识别"不要…""排除…""除了…"等否定语义，转成 `must_not` 过滤条件
- [ ] **多商品对比**：当意图为对比时，并行检索 2–3 件商品，Prompt 模板生成结构化对比表（属性维度由 LLM 总结）
- [ ] **主动澄清**：信息不足时返回 `event: clarify` 让客户端展示 chips 选项（如"拍照优先 / 续航优先 / 性价比优先"）

**验收**：用以下 5 条 query 跑回归测试，全部输出合理：
1. "帮我推荐跑鞋" → "要轻量的" → "预算 500 以内"
2. "200 元以下的蓝牙耳机有哪些？"
3. "对比一下 A 和 B 哪个更保湿"
4. "推荐防晒霜，但我不要含酒精的，也不要日系品牌"
5. "推荐一款手机" → Agent 主动反问

### Phase 5 · 加分项（按优先级选做，5 天）

> 评分原则："做精一项胜过浅尝三项"。建议**选 1 主线 + 1 辅线**深入做。

## 6. 关键设计约束（评审高频考点）

### 6.1 防幻觉（**必须做到**）

- 系统 Prompt 中明确："只能基于下方 `<retrieved_products>` 中出现的 `product_id` 和字段作答；若用户问及未检索到的商品，必须回答'抱歉，库内暂未找到匹配的商品'。"
- 输出商品卡片走**结构化 JSON 事件**，字段必须从检索结果原样取出，禁止 LLM 自由生成 product_id / 价格
- 后端在 SSE 发送 `product_card` 前**做一次校验**：`product_id` 必须在 MySQL `products` 表中存在（用 AsyncSession 单条 SELECT），否则丢弃该卡片

### 6.2 流式低延迟

- 首 token 目标 < 1.5 s（加分项目标 < 1 s）
- 检索和 embedding 用 `asyncio.gather` 并行
- LLM 调用 `stream=True`，每收到 chunk 立刻 yield，不要等整段

### 6.3 错误降级

| 失败点 | 降级策略 |
| --- | --- |
| Doubao 限流 / 超时 | 返回固定话术 + 直接推送检索 Top-3 卡片 |
| Embedding 失败 | 落 BM25 兜底（用 `rank_bm25` 库） |
| Milvus 不可用 | 客户端展示"系统繁忙"，不阻塞 UI |
| MySQL 不可达 | 健康检查降级；卡片校验失败时只展示文本回复 |

### 6.4 安全与合规

- API Key 仅放 `.env`，`.gitignore` 必须包含
- 商品图片在 iOS 端走 `https`（开发期允许 ATS 例外）
- 用户上传图片不持久化（处理完即删）

# 05 · 多模态交互设计（Phase 5）

> 配套主文档 `01_项目开发文档.md`。本篇聚焦 Phase 5 多模态交互加分项：5B「拍照找货」与 5C「语音输入 / TTS 播报」。5B 走 iOS 端拍照/选图 → 服务端 vision embedding → 与 Phase 4 结构化筛选融合的 RAG 检索 → 商品推荐流；5C 走 iOS 端录音/播放 → 服务端豆包语音 OpenSpeech ASR/TTS → 识别文本复用现有 RAG 主链路。
> 数据工程基础见 `02_数据工程与RAG设计.md`，后端 API / Agent 编排见 `03_后端API与Agent编排.md`，iOS 端结构见 `04_iOS客户端设计.md`，本文档不重复这些通用基础。

## 1. 范围与目标

### 1.1 课题对照

| docs/01 Phase 5 加分项 | 本期取舍 |
| --- | --- |
| 5A 业务闭环（购物车/下单） | ❌ 不做。工程量大且不增对话能力。 |
| **5B 多模态拍照找货** | ✅ **主线，纵向做深**。 |
| **5C 语音 / TTS** | ✅ **服务端 ASR/TTS 主链路**。iOS 端只负责录音、上传、播放与本地降级，识别文本仍复用现有 chat/RAG。 |
| 5D 工程性能（缓存/首 token/骨架屏） | ❌ 不做。100 件 demo 上看不出数据，反而显得凑。 |

原则承袭课题"做精一项胜过浅尝三项"：5B 仍是检索能力主线；5C 做成独立音频网关，避免把 ASR/TTS 逻辑塞进 chat/RAG 编排层。

### 1.2 Phase 5 目标

在 Phase 4（对话能力增强）之上叠加多模态输入/输出路径，达成：

1. **iOS 端可选图/拍照**与文字输入并存，上传到服务端
2. **服务端用 vision embedding 把图编码为查询向量**，与 Phase 4 结构化筛选条件（价格 / 品牌排除 / 品类）协同检索
3. **图文矛盾 LLM 仲裁**：retrieve 层不做语义裁决
4. **任意一环失败均能退化为纯文本流**：拍照不是新的必经路径，是增强项
5. **15 条图搜评测黄金集**：四类指标全部达标（详见 §6.3）
6. **服务端 ASR 语音输入**：iOS 录 16k PCM，上传 `/api/v1/audio/asr`，返回文本填入输入框，发送后走同一 chat 请求
7. **服务端 TTS 播报**：assistant 回复可手动/自动调用 `/api/v1/audio/tts`，返回 WAV 后由 iOS 播放
8. **用户可选播报音色**：支持 10 个方舟 voice id，Header 菜单选择，后端白名单校验
9. **语音失败不影响文本与图搜**：服务端 ASR/TTS 失败只提示或降级到本地语音能力，不阻塞原有对话

## 2. 关键决策记录

5B 图搜的四个核心 fork 选型：

| 决策点 | 选型 | 替代方案与放弃理由 |
| --- | --- | --- |
| 输入形态 | **图 + 文混合** | 纯图搜逻辑最简，但与现有过滤能力脱节，看起来像两个独立功能 |
| 图向量入库 | **复用 `products_text` collection**，chunk_type='image' 标量隔离 | 另开 `products_image` collection 多一套 schema/seed/双路 retrieve，100 件规模过度设计 |
| 上传 API | **二段式 `/upload/image` → image_id → `/chat`** | 一段式 `/chat` multipart + SSE 混在一个路由别扭，上传失败要在 SSE 表达制造复杂度 |
| 图文融合检索 | **方案 A：单向量多模态编码 + Phase 4 scalar filter** | B 双向量并行检索 + 加权合并（200+ 行 + 权重玄学，100 件上看不出收益）；C 纯图搜再 text rerank（图未召回的永远进不来） |

**方案 A 的关键洞察**：Doubao `multimodal_embeddings.create()` 接口本来就接 `[{type:text}, {type:image}]` 一起 → 返回**单个 2048 维统一向量**，这是接口契约的目标用法。Phase 4 的 query_rewriter / structured_retriever / scalar filter 全部零侵入复用——只是 query_vector 由 multimodal embed 生成。

5C 语音 / TTS 的核心选型：

| 决策点 | 选型 | 替代方案与放弃理由 |
| --- | --- | --- |
| 语音输入 | **服务端 OpenSpeech ASR**。iOS 用 AVAudioEngine 录 16k/16-bit/mono PCM，上传 `/audio/asr`，后端走 `wss://openspeech.bytedance.com/api/v3/sauc/bigmodel` 二进制 WebSocket 协议 | 端侧 Speech.framework 保留为降级；直连 OpenSpeech 会暴露 API key |
| TTS 播报 | **服务端 OpenSpeech TTS**。后端走 `wss://openspeech.bytedance.com/api/v3/tts/bidirection` V3 双向流式协议，返回 WAV，iOS 用 AVAudioPlayer 播放 | 取消 AVSpeechSynthesizer 原生朗读模式，避免服务端音色和本地音色混用 |
| 与 RAG 关系 | **识别后的文本复用现有 `/chat`** | 为语音单独建 chat 路由会分裂已有文本/图片能力 |
| 自动播报 | **默认关闭，用户可手动打开** | 默认自动播报容易打扰用户，也会在调试和答辩时产生噪音 |
| 音色选择 | **Header 菜单选择，后端白名单校验** | 客户端自由输入 voice id 容易打到未开通音色或产生 4xx |

**5C 的关键洞察**：ASR/TTS 是音频网关，不是 RAG 编排的一部分。服务端统一接豆包语音 OpenSpeech、resource id 和音色，iOS 只处理录音/播放/选择状态；识别后的文本继续复用既有 `/chat`，因此不会分裂文本、图片、多轮记忆链路。

## 3. 架构与数据流

### 3.1 整体

```
┌─────────────────────────  iOS (PriceCat)  ─────────────────────────┐
│  ChatView 输入栏                                                    │
│  ┌─[图]─[麦克风]─[图缩略图×]─┬────[文字/语音转写输入]────┬─[发送]─┐      │
│  Assistant 气泡：回复正文 + 商品卡片 + [播报]按钮；Header 可切自动播报   │
│                                                                    │
│  发送：① 若选图 → POST /upload/image (multipart) → {image_id}       │
│        ② POST /chat {message, image_id} (JSON)                    │
│        ③ 用户气泡：顶部缩略图 + 下方文字气泡                            │
│  语音：① 麦克风按钮 → 录 16k PCM → POST /audio/asr → inputText        │
│        ② 用户确认发送 → 仍按文本/图文 chat 流程处理                       │
│  TTS： assistant done 后手动/自动 POST /audio/tts → WAV → AVAudioPlayer │
└──────────────────────────────┬─────────────────────────────────────┘
                               │ SSE
┌──────────────────────────────▼─────────────────────────────────────┐
│  Server / FastAPI                                                  │
│                                                                    │
│  POST /upload/image                                                │
│  ├─ 校验：≤ 1MB / jpeg|png|webp / Pillow 可解码                      │
│  ├─ 落盘：data/uploads/<yyyymmdd>/<uuid>.jpg                        │
│  ├─ 同步算 vision embedding → ImageEmbedCache.put(image_id, vec)    │
│  └─ 返回 {image_id, preview_url}                                    │
│                                                                    │
│  POST /chat                                                        │
│   └─ orchestrator.handle(message, image_id?, history, summary)     │
│                                                                    │
│  POST /audio/asr                                                   │
│   └─ Volc OpenSpeech ASR → {text}                                    │
│                                                                    │
│  POST /audio/tts                                                   │
│   └─ Volc OpenSpeech TTS → audio/wav                                 │
│       ├─ image_id == None → Phase 4 recommend / compare / clarify  │
│       └─ image_id 非空 → MultimodalBranch.handle():                 │
│            1. query_vec = ImageEmbedCache.get(image_id)             │
│                          ?? embedder.embed_multimodal(img, text)    │
│            2. parsed = query_rewriter.parse(text, history, summary) │
│            3. filter_expr = build_filter(parsed)                    │
│                            AND chunk_type IN ("image","title")      │
│            4. hits = milvus.search(query_vec, filter_expr)          │
│            5. products = retriever._aggregate(hits)                 │
│            6. msgs = build_image_search_messages(...)               │
│            7. LLM 流式 → card_extractor → SSE 9 类事件                │
└────────────────────────────────────────────────────────────────────┘

┌──────────────────────  Vector Index (Phase 5 增量)  ───────────────┐
│  Phase 1 已有：title / description / faq / review chunks            │
│  Phase 5 增量：每件商品 +1 条 image chunk                             │
│     ├─ source_id  = "<pid>#image#0"                                │
│     ├─ chunk_type = "image"                                        │
│     ├─ text       = "<image_path>"   (占位，不进 prompt)             │
│     └─ vector     = embedder.embed_image(<pid>_live.jpg)           │
│  100 chunk 增量 → 总 1192 chunks                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 3.2 关键设计要点

1. **共用 collection**：vision 模型对图文统一编码到同一 2048 维空间，标量字段 `chunk_type` 做隔离即可。
2. **检索路径不分裂**：multimodal branch 仍是**一次 milvus search**，query_vector 多模态编码、filter_expr 多带 `chunk_type IN ('image','title')`。
3. **结构化条件零侵入**：价格 / 品牌 / 品类筛选走 Phase 4 `query_rewriter.parse()`（图本身不携带价格信息，价格永远从文字抽）。
4. **`chunk_type IN ('image','title')` 的意义**：让图搜同时召回标题命中——例如上传"白色洗面奶"图 + 文字"洗面奶"，"洗面奶"靠 title chunk 召回，颜色靠 image chunk。
5. **图文矛盾不在 retrieve 层仲裁**：Prompt 加一条「图文矛盾以文字为准」由 LLM 处理。
6. **`ImageEmbedCache`**：内存 LRU（cap=100、TTL=30min），避免同一 image_id 在 multi-turn 里重复算 vision embedding。冷启动 / 缓存 miss 时从落盘图重算（~+500ms）。
7. **Phase 4 退化路径仍在**：image_id 为空时走原 recommend/compare/clarify——纯文本用户体验零变化。

### 3.3 5C 语音输入与 TTS 数据流

```
┌──────────── iOS ChatView ────────────┐
│ 麦克风按钮                            │
│   ├─ request Microphone auth          │
│   ├─ AVAudioEngine 录音               │
│   ├─ AVAudioConverter → 16k PCM       │
│   ├─ stop 后 POST /api/v1/audio/asr   │
│   └─ ASR text -> inputText            │
│                                       │
│ 发送按钮                               │
│   └─ ChatViewModel.send()             │
│       └─ transport.stream(message, imageID?)  # 原链路不变
│                                       │
│ Assistant 回复                         │
│   ├─ Header 菜单选择 voice             │
│   ├─ 手动点击 speaker                  │
│   └─ 或 autoSpeak 开启且 .done 后播报    │
│       └─ POST /api/v1/audio/tts       │
│           └─ WAV -> AVAudioPlayer     │
└───────────────────────────────────────┘
```

**边界**：
- ASR 返回文本只写入输入框，不自动发送，避免识别误差直接触发请求。
- 手动停止录音后允许远端 ASR completion 回填输入框；发送 / 新建会话会让旧 ASR 回调失效。
- TTS 播报只读 assistant 文本，不读商品卡片 JSON 或图片路径。
- resetSession 必须停止录音与播报，避免旧会话声音继续播放。
- 服务端 ASR 失败时，iOS 仍可退回端侧 Speech.framework；服务端 TTS 失败时不再走 AVSpeechSynthesizer 原生朗读，只停止本次播报。

### 4.3 设计原则注脚

- **`multimodal_branch.py` 独立成文件**：避免 orchestrator.py 进一步膨胀。它和 `clarify_detector` / `compare_planner` 同层级——都是"特定意图的处理器"。
- **`ImageEmbedCache` 独立成模块**：要被 `upload.py` 和 `multimodal_branch.py` 同时依赖，放在 retriever.py 里两边 import 会交叉。
- **`build_image_index.py` 不并入 `build_index.py`**：Phase 5 索引行为是"增量补 image chunk"，分开方便单独跑 `--rebuild` 不影响文本 chunk。
- **音频 API 独立在 `/audio` router**：ASR/TTS 是输入输出网关，不进入 orchestrator；Chat 只消费 ASR 后的文本。

## 5. 错误降级矩阵

| 失败点 | 检测 | 降级行为 | 用户体验 |
| --- | --- | --- | --- |
| 上传 MIME 不合法 | upload.py 校验 | HTTP 415 + `{error:"图片格式不支持"}` | iOS toast 提示重选 |
| 上传 > 1MB | upload.py 校验 | HTTP 413 | iOS toast；客户端在 ImagePicker 已先压缩，正常不触发 |
| 图解码失败 | Pillow open() except | HTTP 422 | iOS toast"图片无法读取" |
| Doubao vision 限流/超时 | tenacity 重试 3 次仍失败 | upload 返 503 + `{degraded:true, fallback_text_only:true}` | iOS 提示"图片识别繁忙，可用文字继续"——chat 仍能发（不带 image_id） |
| `ImageEmbedCache` miss（重启后） | get() 返回 None | multimodal_branch 兜底：读落盘图重 embed 一次（~+500ms） | 用户无感，首 token 略增 |
| 落盘图被清理（image_id 悬空） | open() 抛 FileNotFoundError | 退化到纯文本流，emit warning 事件 | iOS 提示"图片已失效，按文字处理"，仍正常出推荐 |
| Milvus search 空结果 | hits == [] | 走 Phase 4 现有 retriever.search(text only) 兜底 | 用户至少能看到文字命中结果，不会"上传图就空" |
| LLM 流式失败 | sse 异常 | Phase 2 已有降级：吐 Top-3 retrieved 卡片 | 与现有一致 |
| 麦克风权限拒绝 | Microphone auth denied | `voiceNotice` 提示"麦克风权限未开启" | 文本输入、图片上传、发送按钮照常可用 |
| `/audio/asr` 503 | OpenSpeech ASR 失败 / WebSocket 异常 / 401 鉴权失败 | iOS 提示"语音识别失败"，保留手动输入；可退回端侧 Speech.framework | 用户可继续手动输入 |
| `/audio/tts` 503 | OpenSpeech TTS 失败 / voice 未开通 / 401 鉴权失败 | `ServerSpeechSynthesisService` 停止本次播报，不走本地朗读 | 自动/手动播报不阻塞聊天 |
| unknown voice | 后端白名单校验 | HTTP 422 | iOS 只展示内置菜单，正常不触发 |
| 用户发送时仍在录音 | `send()` 前检查 | 先停止录音，再按当前 inputText 发送 | 避免录音和发送状态交叉 |
| 新建会话时正在播报 | `resetSession()` | 停止 TTS 播报并关闭录音 | 旧会话声音不会延续到新会话 |
| assistant 回复为空 | 点击播报 / 自动播报 | 不调用 TTS | 无多余提示 |

**核心兜底原则**：图任意一环出错，**始终能退化为纯文本流**。绝不出现"上传图后整个对话卡死"。

## 8. YAGNI 边界（明确不做）

- ❌ 图片 EXIF 旋转处理（iOS PhotosPicker 已自动正向）
- ❌ 图片 OCR 抽文字喂检索（vision embedding 本身已学到字形语义，纯 OCR 增益小且复杂）
- ❌ 多图上传（一次一张已够 demo）
- ❌ 上传图持久化超 24h（每日 cron 清理即可，Phase 6 再做）
- ❌ 历史图复用（"还按第 1 轮那张图"心智复杂，Phase 5 不接）
- ❌ compare 分支接图（demo 用例少，Phase 6 再说）
- ❌ iOS 端直连方舟 ASR/TTS（API key 不能暴露给客户端）
- ❌ 自由输入 voice id（只允许内置白名单，避免调用未开通音色）
- ❌ 语音自动发送（识别误差会直接触发错误请求，必须由用户确认发送）
- ❌ 播报商品卡片结构化字段（TTS 只播 assistant 文本正文）
